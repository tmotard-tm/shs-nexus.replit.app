Sprint 2: Reprovisioning Checklist for Phone Recovery
I'm the PM for the Technician Offboarding Process Redesign. This is Sprint 2 of the Phone Recovery Queue feature. Sprint 0 (schema) and Sprint 1 (contact logging components) are complete.

Context
Stage 1 (Recovery) components are working. Team members can contact departing techs, track shipping labels, and mark phones as received. When a phone is marked "Received," it transitions to Stage 2.

This sprint builds the specialist's reprovisioning checklist component. One person currently handles all phone reprovisioning: receiving phones at the corporate office, wiping them, reprovisioning for new hires, and reinstating service. This replaces their personal spreadsheet.

Important architectural note: Like Sprint 1, this sprint builds a reusable component that Sprint 3 will import into the slide-out detail panel. Do NOT create a separate page or route.

Note: The full reprovisioning checklist is still being gathered from the specialist. The steps below are based on a discovery session and cover the known workflow. Build the component to be extensible so additional sub-steps can be added later without restructuring.

What You're Building
One reusable React component plus supporting API routes:

ReprovisioningChecklist - A vertical progress tracker with 5 steps
ReprovisioningChecklist Component
Location: client/src/components/phone-recovery/

A vertical progress tracker with checkboxes and expandable detail fields. Use purple accent color for Stage 2 elements. A visual progress line fills as steps are completed.

Step 1: Receive & Inspect

Physical condition (dropdown): Good, Damaged, Non-functional
Condition notes (textarea, shown if Damaged or Non-functional selected)
If "Non-functional" is selected, show a "Mark as Write-Off" button
Step 2: Data Wipe

Data wipe completed (checkbox)
Wipe method (dropdown): Factory Reset, Secure Erase, MDM Remote Wipe
Step 3: Reprovision

Reprovision completed (checkbox)
Carrier/line details (text, optional)
Step 4: Reinstate Service

Service reinstated (checkbox)
Step 5: Assign to New Hire

Assigned to new hire (text, optional): Name or employee ID
New hire department (text, optional)
Date ready: Auto-set to current timestamp when all of the above checkboxes (wipe, reprovision, service) are checked
Each step should have a visual indicator showing completion. The progress line should fill proportionally as steps are completed.

Status Flow
Create a utility function deriveReprovisioningStatus(task) that Sprint 3 can import:

Received (phoneDateReceived set, nothing else done)
  -> Inspecting (phonePhysicalCondition is set)
    -> Wiping (phoneDataWipeCompleted = true)
      -> Reprovisioning (phoneReprovisionCompleted = true)
        -> Ready for Deployment (phoneServiceReinstated = true)
          -> Assigned (phoneAssignedToNewHire is set)
Non-Functional Device Path
If phonePhysicalCondition = "Non-functional":

Show a "Mark as Write-Off" button prominently
On click: set phoneWrittenOff = true
Task is removed from the active pipeline (written-off devices should still be queryable)
API Routes
Add to server/routes.ts:

PATCH /api/phone-recovery/:id/inspect - Set physical condition and condition notes
PATCH /api/phone-recovery/:id/reprovisioning - Update wipe, reprovision, service, and assignment fields
PATCH /api/phone-recovery/:id/write-off - Mark device as written off
Constraints
Build as a reusable component in client/src/components/phone-recovery/
Do NOT create a new page or route
Do NOT build the dashboard page yet (Sprint 3)
Build the checklist to be extensible (additional steps may be added later)
Each checkbox should persist independently (saving one step doesn't require completing all steps)
Export deriveReprovisioningStatus() utility function
Follow existing Nexus UI patterns for checklist-style forms
Acceptance Criteria
 ReprovisioningChecklist component with all 5 steps renders correctly
 Each checkbox persists independently on save
 Progress line fills as steps are completed
 deriveReprovisioningStatus() correctly derives status from checklist state
 Non-functional devices can be written off via "Mark as Write-Off" button
 Written-off devices set phoneWrittenOff = true
 "Assigned to new hire" and "New hire department" fields work
 Date ready auto-sets when all checkboxes are checked
 Wipe method is a dropdown (Factory Reset / Secure Erase / MDM Remote Wipe)
 Component is importable from client/src/components/phone-recovery/
 API routes work and validate input
