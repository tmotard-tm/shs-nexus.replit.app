1. The "Dual-Write" Gap
Risk: Between the time you start the migration script (Task 3) and the moment you deploy the redirect (Task 5), users or automated processes (like the Samsara GPS integration mentioned in Task 2) might write new data to the original Fleet-Scope database.

Fix: Add a "Maintenance Mode" step for the original app. Before running the final migration script, the original app should be put into a read-only state or show a "Migrating - Please Wait" banner to ensure no data is lost in the final minutes.

2. Table Name Collision & Namespace Logic
Risk: You noted that shared/fleet-scope-schema.ts defines tables without an fs_ prefix. While this makes the code migration easier, it creates a high risk of "Namespace Pollution."

Gap: If Nexus ever needs its own trucks or actions table for a non-Fleet-Scope feature, you will have a naming conflict.

Recommendation: Even if you don't rename the tables now, you should ensure the Drizzle/SQL schema is logically separated (e.g., putting Fleet-Scope tables into a specific Postgres schema like fleet_scope.trucks rather than the default public.trucks).

3. Identity & Sequence Sync
Risk: If your PostgreSQL tables use SERIAL or GENERATED BY DEFAULT AS IDENTITY for primary keys, copying data via INSERT often doesn't update the underlying sequence counter.

Gap: After the switch, the first time Nexus tries to create a new record (e.g., a new PO or truck), the DB might try to use an ID that already exists, causing a unique constraint violation.

Fix: Add a "Sequence Reset" step to your migration script to update all table sequences to MAX(id) + 1.

4. Reversibility (The Rollback Plan)
Risk: If Task 4 (The Switch) reveals a critical bug in the Nexus Fleet-Scope module that wasn't caught in Phase 1, you need a way to go back.

Gap: Once you issue a 301 (Permanent Redirect), browsers cache that instruction. Reversing it is difficult.

Recommendation: Use a 302 (Found/Temporary) redirect for the first 24–48 hours. Only switch to a 301 (Permanent) once you have verified data is flowing correctly into the Nexus DB without errors.

5. Dependency on FS_DATABASE_URL
Risk: The plan removes FS_DATABASE_URL from Nexus at runtime (Task 4), but several background tasks (Samsara GPS, AI call logs, PMF sync) may rely on external API keys or environment variables that were only present in the original Replit.

Gap: Ensure all third-party secrets (Samsara API keys, AI credentials) are ported from the Fleet-Scope secrets to the Nexus secrets.
