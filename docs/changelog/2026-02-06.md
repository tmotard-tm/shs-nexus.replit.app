# Changelog: 2026-02-06

## Bug Fixes

### 1. Communication Template Seeding Fix
- **Problem**: `seedDefaultTemplates()` skipped seeding if ANY templates existed in DB, so `tool-audit-notification`, `credit-card-deactivation`, and `password-reset` were never created
- **Fix**: Changed to check individual template names and only create missing ones
- **Impact**: Templates now seeded correctly on startup even when other templates exist

### 2. Foreign Key Constraint Error
- **Problem**: `sentBy: 'system'` passed to `communication_logs` insert violated FK constraint (references `users.id`)
- **Fix**: Changed to `sentBy: undefined` in `notification-service.ts` for system-triggered notifications, which becomes `null` via `|| null` in the service
- **Impact**: Communication logs now save correctly for automated notifications

### 3. Startup Template Seeding
- Added `seedDefaultTemplates()` call to `server/index.ts` startup sequence
- Runs after workflow template seeding, before Snowflake initialization
- Non-blocking: failures don't prevent server start

### Files Modified
- `server/communication-service.ts` - Fixed seeding logic to check individual templates
- `server/notification-service.ts` - Changed `sentBy` from `'system'` to `undefined`
- `server/index.ts` - Added communication template seeding at startup

   - Added to `server/notification-service.ts`

   - Logs results for each recipient
   - Handles errors gracefully

3. **Integration into Phase 2 Trigger**
   - Added to `createPhase2FleetTasks()` in `server/storage.ts`
   - Fires after Phase 2 Fleet tasks are successfully created
   - Non-blocking: notification failures don't prevent task creation

4. **Configurable Recipients**
   - Primary: `PHASE2_NOTIFICATION_RECIPIENTS` environment variable (comma-separated emails)
   - Fallback: Users with Fleet department assignment in the database
   - Logs a message if no recipients found (graceful degradation)

### Files Modified

- `server/notification-service.ts` - Added `sendPhase2TasksCreatedNotification()` function

2. **Switch mode**: In Communication Hub, find `phase2-tasks-created` template and change mode from `simulated` to `whitelisted` or `live`

## Tools Queue Duplicate Task Fix

### Problem
Two separate systems were creating tasks for the same terminated employees:
1. **Snowflake sync service** created sparse "Day 0: Recover Equipment & Tools" tasks
2. **GET /api/tools-queue handler** auto-created rich "Tools Queue -" tasks as a side effect every time someone loaded the page

Both systems had incomplete duplicate detection — each only checked for its own format, not the other's.

### What We Fixed
1. **Consolidated task creation**: Moved all Tools task creation into `snowflake-sync-service.ts` (the sync service). GET handler now only reads data.
2. **Rich format by default**: Sync service now creates "Tools Queue -" format tasks with full HR separation data (employee name, ID, separation date, truck number, pickup address) instead of sparse Day 0 format.
3. **Cross-format duplicate detection**: Updated `checkOffboardingDuplicates()` and enrichment logic in `routes.ts` to check both `employee.*` and `technician.*` data structures, preventing re-creation regardless of which format an existing task uses.
4. **Database cleanup**: Removed 4 duplicate Day 0 tasks, keeping the richer "Tools Queue -" versions.
5. **Dev database refresh**: Ran `scripts/refreshDevFromProd.js` to sync dev DB from prod (32 tables).

### Design Decision
- GET handlers should only read data, never create it
- Single source of truth for Tools task creation prevents race conditions
- Fleet and Inventory queues still use Day 0 format (unchanged)

### Files Modified
- `server/snowflake-sync-service.ts` — Tools tasks now created in "Tools Queue -" format with HR data; email notifications integrated
- `server/routes.ts` — Removed auto-creation from GET handler; updated duplicate detection for both data formats
